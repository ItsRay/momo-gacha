# 📦 【momo 品牌狂歡】整點幸運轉盤系統 - 產品需求文件 (PRD)

## 1. 產品目標與核心功能

打造一套支援瞬間高併發的 **「整點即時幸運轉盤系統」**，提升 App 活躍度，同時確保行銷預算絕對安全。

* **營運端 (Admin)**：可自訂活動檔期與獎項發行庫存。支援 **動態調整獎項權重**，無須重啟服務即刻生效。
* **用戶端 (Client)**：即時開獎。中獎後透過背景任務觸發中獎紀錄與相關派發流程，確保抽獎主流程極致順暢。

## 2. 系統效能指標與流量預估 (Capacity Planning)

為因應大型電商檔期（如 618）的整點搶抽熱潮，系統需滿足以下非功能性需求：

* **峰值流量預估 (Peak QPS)**：假設整點行銷推播發送後，觸發「羊群效應」，預期 10 萬名活躍用戶於 10 秒內瞬間湧入點擊。系統需穩定承載核心 API **10,000 QPS 的瞬間寫入 (Write) 請求**。
* **API 回應時間 (Latency)**：用戶端抽獎請求（不含網路傳輸）的 P99 延遲必須 **小於 1s**。
* **服務可用性 (Availability)**：目標 99.99%，且具備極端情況下的降級保護能力。

## 3. 核心商業邏輯：預算控管與兜底機制 (Fallback)

為保護行銷預算，各獎項「命中機率」為固定值。系統具備嚴格的庫存卡控與 **「無限量兜底替換」** 機制。

營運只需設定限量獎項的機率（例：iPhone 1%、折價券 10%），系統會自動將剩餘的機率空間分配給「無限量兜底獎（如：銘謝惠顧）」。當限量大獎耗盡時，底層機制會自動降級派發兜底獎，確保無客訴風險且預算不超支（詳細情境請見第 5 點範例）。

## 4. MVP 範圍收斂 (Scope)

為聚焦展示「高併發處理」與「非同步架構」，本專案收斂以下邊界：

* 透過 API Header `X-User-Id` 進行身分 Mock，略過完整會員登入（架構設計上預期由最前線的 API Gateway 負責進行 Token 集中驗證，驗證成功後再將用戶 ID 以 `X-User-Id` Header 明碼向下游微服務傳遞）。
* 略過 DB Transaction 實際資產入帳，專注於「抽獎當下的庫存防護」與「事件觸發」。

## 5. 附錄：具體情境範例說明

假設營運設定了以下活動，系統自動補滿兜底機率為 89%（100% - 1% - 10%）：

* 【限量大獎】iPhone 17 Pro（機率 1%，總發行量 2 台）

* 【限量二獎】100 元折價券（機率 10%，總發行量 100 張）

* 【無限量兜底獎】銘謝惠顧 / 無獎品（機率 89%，無限量）

* **情境 A（庫存充足時）**：使用者小明抽獎，落入 1% 的機率區間，系統確認 iPhone 還有庫存，小明順利抽中 iPhone。

* **情境 B（大獎耗盡時防超賣）**：iPhone 的 2 台已被抽完。使用者小華抽獎時，若機率運算再次命中這 1% 的大獎區間，基於預算保護原則，系統將 **自動派發「兜底獎（銘謝惠顧）」**。

* **商業與 UX 價值**：前端轉盤畫面將直接順暢地呈現抽中「銘謝惠顧」，使用者無剝奪感（確保 0 客訴）；同時避免大獎耗盡導致二獎機率暴增，完美鎖住行銷預算。
